# 05_benchmark_plan
## 1. 문서 목적

이 문서는 baseline 서버의 성능과 안정성을 어떤 기준으로 측정할지 정의한다.

baseline benchmark의 목적은 다음과 같다.

- 현재 baseline의 기준 성능을 기록한다
- 병목이 어디서 발생하는지 파악한다
- 이후 고도화 버전과 비교할 기준을 만든다
- correctness를 유지하면서 어느 정도 성능을 내는지 확인한다

---

## 2. 측정 원칙

baseline 단계의 benchmark는 단순히 빠른지를 확인하는 것이 아니라 correctness를 유지하는 현재 구조의 성능 특성을 파악하는 데 목적이 있다.

따라서 다음 원칙을 따른다.

- 처리량과 응답 시간을 함께 본다
- 동시 클라이언트 수 증가에 따른 변화를 본다
- workload에 따라 성능 차이가 있는지 본다
- 성능 수치와 함께 불변조건 유지 여부를 확인한다
- 병목이 lock인지, 영속 기록인지 확인하려고 한다

---

## 3. 핵심 측정 항목

baseline 단계에서는 다음 항목을 우선적으로 측정한다.

- 처리량
  - 단위 시간당 처리한 요청 수
  - 예: requests/sec

- 평균 응답 시간
  - 요청 1개를 처리하는 데 걸린 평균 시간

- 동시 클라이언트 수에 따른 성능 변화
  - 클라이언트 수를 늘렸을 때 처리량과 평균 응답 시간이 어떻게 변하는지 측정한다

- workload 유형별 성능
  - read-heavy
  - write-heavy
  - mixed workload

- append-only log 기록 시간
  - 성공한 상태 변경 요청 1건을 로그에 기록하고 durable 반영하기까지 걸리는 시간

- lock 대기 시간
  - worker thread가 전역 mutex를 얻기까지 기다린 시간

- 불변조건 검증 결과
  - 재고 음수 발생 여부
  - lost update 발생 여부
  - 응답과 실제 상태 불일치 여부

---

## 4. benchmark 시나리오

### 4.1 단일 클라이언트 기준 측정

가장 기본적인 처리량과 평균 응답 시간을 측정한다.

### 4.2 동시 클라이언트 증가 실험

클라이언트 수를 늘리면서 처리량과 평균 응답 시간 변화를 측정한다.

예:
- 1 client
- 10 client
- 50 client
- 100 client

### 4.3 workload 유형별 실험

다음 workload를 구분해서 측정한다.

- read-heavy
- write-heavy
- mixed workload

### 4.4 hotspot 경쟁 상황 실험

여러 클라이언트가 동일한 stock에 집중적으로 접근하는 상황을 측정한다.  
이를 통해 lock contention과 직렬화 영향을 확인한다.

---

## 5. 결과 해석 기준

benchmark 결과는 다음 기준으로 해석한다.

- 처리량이 높아도 불변조건이 깨지면 올바른 결과로 보지 않는다
- 평균 응답 시간은 workload와 동시 클라이언트 수 변화와 함께 해석한다
- append-only log 기록 시간이 길다면 영속화 비용이 병목일 수 있다
- lock 대기 시간이 길다면 전역 mutex가 병목일 수 있다

---

## 6. 선택 측정 항목

다음 항목은 baseline 1차 측정에서는 필수는 아니지만, 여유가 있으면 추가로 측정할 수 있다.

- 명령별 응답 시간
  - `show`
  - `buy`
  - `sell`

- 서버 시작 및 복구 시간

- queue 대기 시간

- CPU 사용량
- 메모리 사용량
- 디스크 쓰기 비용

- 장시간 실행 안정성

---

## 7. 한 줄 요약

baseline benchmark는 단순 성능 측정이 아니라, correctness를 유지하는 현재 구조의 기준 성능과 병목 위치를 기록하기 위한 계획이다.
