# 00_project_overview
## 1. 이 프로젝트는 뭘까?

이 프로젝트는 동시 접속 환경에서 여러 클라이언트의 요청을 안전하고 효율적으로 처리하는 concurrent stock server를 구현하고 concurrency model에 따른 차이를 비교 및 분석하는 시스템 프로젝트다.  
  
단순히 요청을 처리하는 서버를 만드는 것이 아니라 공유 상태를 가지는 거래 시스템에서 동시성을 어떻게 제어하고 어떤 설계가 더 적절한지를 구현과 실험을 통해 검증하는 것을 목표로 한다.

---

## 2. 이 프로젝트의 목표

이 프로젝트의 목표는 다수의 클라이언트가 보내는 요청을 다음 기준 아래에서 처리할 수 있는 서버를 설계하고 구현하는 것이다.

- 안전성
- 일관성
- 효율성

즉, 이 프로젝트는 단순 기능 구현을 넘어 동시성 환경에서 공유 데이터를 다루는 서버가 어떤 기준을 만족해야 하는지 정의하고 이를 실제 시스템으로 검증하는 데 초점을 둔다.

---

## 3. 이 프로젝트의 핵심 기능

### 3.1 다중 클라이언트 연결 처리

여러 클라이언트가 동시에 접속해서 요청을 보내더라도 서버는 계속 정상 동작해야 한다.

이 과정에서 다음과 같은 질문을 다룬다.

- 동시에 여러 연결을 안정적으로 처리할 수 있는가?
- 특정 클라이언트 하나 때문에 전체 서버가 막히지 않는가?
- 서버 장애 상황을 고려해 데이터 백업이나 복구 전략이 필요한가?

---

### 3.2 공유 데이터의 동시 접근 제어

여러 클라이언트가 같은 종목에 대해 동시에 `buy`/`sell` 요청을 보내면 race condition이 발생할 수 있다.

따라서 서버는 다음과 같은 문제를 제어할 수 있어야 한다.

- 동시 읽기 허용 범위 결정
- 쓰기 요청의 원자성 보장
- 읽기와 쓰기 충돌 제어
- lost update 방지
- invalid state 방지

---

### 3.3 거래 단위 일관성 보장

하나의 요청은 단순한 한 줄 명령처럼 보이더라도 내부적으로는 여러 단계의 상태 전이로 이루어진다.

예를 들어 `buy`는 한 줄 명령이지만 내부적으로는 다음 단계로 이루어진다.

1. 종목 조회
2. 수량 검증
3. 재고 차감
4. 결과 응답

이 4개 단계가 하나의 거래 단위로 움직여야 하며 중간 단계의 불완전한 상태가 외부에 노출되어선 안 된다.

---

### 3.4 성능과 정확성의 균형

동시성 서버는 다음 두 극단으로 무너지기 쉽다.

- 정확성만 챙기면 lock 범위가 너무 커져 성능이 떨어질 수 있다.
- 성능만 챙기면 데이터 일관성과 안정성이 깨질 수 있다.

따라서 이 프로젝트는 정확성을 해치지 않으면서 병렬성과 처리 성능을 최대한 확보하는 방향을 목표로 한다.

---

## 4. 접근 단계

이 프로젝트는 다음과 같은 단계로 진행한다.

### 4.1 안정성 최우선의 baseline 설계

가장 먼저 단순하고 검증하기 쉬운 구조를 기준선으로 설계하고 구현한다.  이 단계의 우선순위는 최고 성능이 아니라 정확성과 안정성 확보에 있다.

### 4.2 안정성이 검증된 뒤 병목 파악

baseline의 안정성이 검증된 이후 실제 실행 환경에서 어떤 부분이 병목이 되는지 측정하고 분석한다.

### 4.3 안전한 범위 안에서 점진적 최적화

병목 지점을 확인한 뒤 안전성을 해치지 않는 범위에서 concurrency model, locking strategy, persistence strategy 등을 점진적으로 개선한다.
