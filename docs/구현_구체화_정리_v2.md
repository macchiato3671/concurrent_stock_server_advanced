# 구현 구체화 정리 v2

## 1. 문서 목적

이 문서는 baseline 구현에 들어가기 전에 현재까지 확정된 구현 방향을 정리한 문서다.  
목표는 설계를 다시 늘리는 것이 아니라, 구현에 바로 들어갈 수 있도록 핵심 선택을 고정하는 것이다.

## 2. baseline 구현 목표

현재 구현의 목표는 다음과 같다.

1. correctness 확보
2. 성공 응답과 실제 상태 일치
3. 성공한 상태 변경의 영속 반영
4. 재시작 후 복구 가능성
5. 이후 benchmark 가능한 구조 확보

## 4. 전역 상태 저장소

### 자료구조
- baseline에서는 기존 BST를 유지한다

### 이유

- 기존 코드와 연속성이 있다
- 전체 조회 시 BST 순회가 자연스럽다
- baseline에서는 최적화보다 설명 가능성과 안정성이 더 중요하다

### stock 항목 필드

- id
- left_stock
- price

### 동작 방식

- 특정 stock 조회: id 기준 BST 탐색
- 수량 수정: 해당 노드 탐색 후 left_stock 변경
- 전체 조회: BST 순회
- 초기 상태 로드: snapshot 또는 초기 stock 파일 기준

---

## 5. 동기화 방식

### 기본 정책

- 전역 상태 저장소는 global mutex 하나로 보호한다
- `show`, `buy`, `sell` 모두 동일한 mutex 아래에서 처리한다

### 상태 변경 요청의 커밋 구간

1. lock 획득
2. stock 조회
3. 유효성 검증
4. 메모리 상태 변경
5. append-only log 기록
6. flush / fsync 등 durable 처리 확인
7. 성공 응답 확정
8. unlock

### 로그 기록 실패 시 롤백

1. old_value 저장
2. 메모리 상태 변경
3. log write + flush / fsync 시도
4. 실패하면 old_value로 복구
5. 실패 응답 반환

## 6. thread pool / queue

### 기본 파라미터

- worker thread 수: 8
- bounded queue 크기: 64

### queue 구조

- 배열 기반 bounded buffer 사용
- `slots`: 빈 공간 수 관리
- `items`: 대기 connfd 수 관리
- `mutex`: queue 수정 보호

### producer / consumer

- producer: main thread가 `accept()`한 connfd를 queue에 넣는다
- consumer: worker thread가 queue에서 connfd를 꺼내 처리한다

### queue full 정책

- baseline에서는 producer가 block하여 빈 자리가 날 때까지 기다린다

### worker 종료 방식

- worker thread는 계속 살아 있으며 queue에서 connfd를 반복해서 가져온다
- 서버 종료 시 poison pill을 queue에 넣고 `pthread_join`으로 정리한다

---

## 7. 네트워크 / 요청 처리 구조

### 처리 단위

- accepted connection(`connfd`) 단위 처리
- 한 연결에서 여러 요청을 반복 처리 가능
- EOF / idle timeout / I/O error 발생 시 해당 연결 종료

### I/O 방식

- CS:APP의 Robust I/O 기반 buffer 사용
- `rio_readinitb`
- `rio_readlineb`
- connfd 단위 오류 처리를 위한 write wrapper

### worker의 연결 처리 흐름

1. connfd 획득
2. `rio_readinitb`
3. `rio_readlineb`로 요청 1줄 반복 읽기
4. 요청 파싱 및 검증
5. 명령 처리
6. 응답 전송
7. `close(connfd)`
8. 다음 connection 대기

## 8. 프로토콜

### 지원 명령

- `show`
- `show <id>`
- `buy <id> <cnt>`
- `sell <id> <cnt>`

### 파싱 규칙

- 공백 기준 split
- 토큰 수로 명령 판정
- 정수 변환 실패 시 `invalid command`
- 음수 또는 0 수량은 `invalid command`

### 응답 형식

- `success`
- `invalid command`
- `stock not found`
- `not enough left stock`
- `internal error`

### `internal error` 사용 기준

- 유효한 요청이었지만 로그 기록 실패, flush / fsync 실패, 파일 I/O 실패 등으로 성공 완료를 보장할 수 없는 경우

### show 출력 포맷

- 전체 조회: 각 줄에 `<id> <left_stock> <price>`
- 단건 조회: `<id> <left_stock> <price>`

예시:
- `show` → 여러 줄 출력
- `show 3` → `3 80 9000`

## 9. 영속화 방식

### baseline 영속화 전략

- snapshot + append-only log

### snapshot

- 초기 stock 파일과 동일한 텍스트 포맷 사용
- 정상 종료 시 1회 저장

### append-only log

- 성공한 `buy`, `sell`만 기록
- 한 줄 포맷:
  - `BUY <id> <cnt>`
  - `SELL <id> <cnt>`

### log write 정책

- 서버 시작 시 log 파일을 append 모드로 연다
- 성공한 상태 변경마다 append write 한다
- 성공 응답 전 flush / fsync로 durable 처리 여부를 확인한다
- 실패하면 rollback 후 failure 응답

### snapshot 이후 log 처리

- snapshot 저장이 성공하면 append-only log는 truncate 하거나 새 파일로 교체한다
- 그렇지 않으면 다음 실행 시 같은 로그를 다시 replay하여 중복 반영될 수 있다

---

## 10. 복구 방식

### 복구 절차

1. snapshot 파일이 있으면 snapshot 파일을 읽는다
2. snapshot 파일이 없으면 초기 stock 파일을 기준 상태로 사용한다
3. 전역 상태 저장소를 구성한다
4. append-only log를 순서대로 replay한다
5. 최신 상태를 복원한다
6. listen socket 준비, worker 생성 및 accept 시작

### 복구 실패 정책

- snapshot 파싱 실패: 서버 시작 실패
- log 형식 오류: baseline에서는 보수적으로 서버 시작 실패
- 복구 완료 판단: snapshot 로드(또는 초기 stock 로드) + log replay 성공 후 listen socket / worker / accept 시작

## 11. 오류 처리 원칙

### 실패로 처리할 상황

- 잘못된 명령
- 없는 stock
- 수량 부족
- log 기록 실패
- snapshot / log 파일 처리 실패

### 원칙

- 실패한 요청은 상태를 변경하지 않는다
- 상태를 이미 변경했다면 성공 응답 전에 롤백한다
- 치명적 오류는 서버 시작 실패 또는 종료로 처리한다
- 비치명적 오류는 실패 응답 후 다음 요청 처리 가능 상태를 유지한다

```text
src/
```

```text
  stockstore.c
  protocol.c
  persistence.c

include/
  stockstore.h
  protocol.h
  persistence.h

  stock.snapshot
```

### 추천

- 처음에는 **최소 구조**로 시작
- 코드가 길어지면 `server/`, `store/`, `persistence/`, `protocol/` 단위로 분리
- `stockclient.c`, `multiclient.c`는 `src` 밖의 `client/`로 빼는 편이 깔끔하다
- `stock.txt`는 `data/stock_init.txt`로 이름을 바꾸는 편이 역할이 더 분명하다

## 17. 한 줄 정리

현재 baseline 구현은 BST 기반 전역 상태 저장소를 global mutex로 보호하고, worker thread pool이 connection 단위로 `show`, `buy`, `sell` 요청을 반복 처리하며, 성공한 상태 변경은 append-only log에 durable 반영한 뒤에만 성공 응답을 반환하고, 재시작 시 snapshot 또는 초기 stock 파일 + log replay로 복구하는 구조로 진행한다.
