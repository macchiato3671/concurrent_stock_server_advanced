# 01_requirements
## 1. 문서 목적

이 문서는 concurrent stock server가 반드시 만족해야 하는 기능 요구사항과 핵심 시스템 성질을 정의한다.

이 문서의 목적은 다음과 같다.

- 서버가 어떤 동작을 해야 하는지 명확히 정의한다.
- 구현 방식과 무관하게 반드시 지켜야 할 조건을 정리한다.
- 이후 baseline 구현과 고도화 버전을 평가할 기준을 제공한다.

---

## 2. 기능 요구사항

서버는 최소한 다음 명령을 지원해야 한다.

- `show`
- `buy <stock_id> <quantity>`
- `sell <stock_id> <quantity>`

각 명령은 다음을 만족해야 한다.

- `show`
  - 현재 stock 상태를 조회할 수 있어야 한다.
  - 조회 결과는 항상 일관된 상태를 기준으로 해야 한다.

- `buy`
  - 특정 stock의 수량 감소 요청을 처리할 수 있어야 한다.
  - stock이 존재하고 수량이 충분한 경우에만 성공해야 한다.
  - 실패한 경우 상태를 변경해서는 안 된다.

- `sell`
  - 특정 stock의 수량 증가 요청을 처리할 수 있어야 한다.
  - 유효한 요청인 경우에만 성공해야 한다.
  - 실패한 경우 상태를 변경해서는 안 된다.

또한 서버는 여러 클라이언트의 동시 접속을 처리할 수 있어야 하며, 동시 요청 상황에서도 아래의 일관성 규칙을 유지해야 한다.

---

## 3. 핵심 불변조건

다음 조건은 시스템이 항상 유지해야 하는 핵심 불변조건이다.

- 모든 stock의 수량은 항상 0 이상의 정수여야 한다.
- 상태 변경 요청은 원자적으로 처리되어야 한다.
- 실패하거나 유효하지 않은 요청은 서버 상태를 변경해서는 안 된다.
- 모든 읽기 요청은 일관된 커밋 상태만 관측해야 한다.
- 클라이언트에게 반환한 응답은 실제 서버 상태와 일치해야 한다.

즉, 하나의 요청은 전부 성공하거나 전부 실패해야 하며, 중간 상태가 외부에 노출되어서는 안 된다.

---

## 4. 성공 응답의 의미

이 시스템에서 성공 응답은 단순히 메모리 값이 잠시 바뀌었다는 뜻이 아니다.

성공 응답은 다음 조건이 모두 만족된 경우에만 반환할 수 있다.

- 요청 형식이 올바르다.
- 대상 stock이 존재한다.
- 필요한 검증을 통과했다.
- 메모리 내 상태 변경이 완료되었다.
- 정의된 영속 저장소 반영이 완료되었다.

즉, 성공 응답은 처리 시작이 아니라 정의된 상태 전이와 영속 반영이 완료되었음을 의미해야 한다.

---

## 5. 영속성 및 복구 요구사항

서버는 다음 요구사항을 만족해야 한다.

- 성공 응답을 반환한 상태 변경 요청의 결과는 유실되어서는 안 된다.
- 성공 응답을 반환한 상태 변경 요청은 영속 저장소에 기록되어야 한다.
- 서버는 재시작 시 이전의 커밋된 상태를 복구할 수 있어야 한다.
- 복구 이후 조회 결과는 장애 이전에 성공적으로 반영된 상태와 일관되어야 한다.

즉, 성공 응답한 상태 변경은 읽기와 재시작 이후에도 계속 유지되어야 한다.

---

## 6. 동시성 요구사항

서버는 동시 요청 환경에서 다음을 만족해야 한다.

- 동일한 stock에 대한 동시 요청으로 lost update가 발생해서는 안 된다.
- 읽기와 쓰기 요청이 겹치더라도 inconsistent read가 발생해서는 안 된다.
- 서버 상태는 단일 기준 저장소를 중심으로 관리되어야 한다.

결과적으로 서로 충돌하는 상태 변경 요청은 어떤 일관된 순서로 처리된 것처럼 설명 가능해야 한다.

---

## 7. baseline 범위

현재 baseline 단계에서는 다음을 가정한다.

- 단일 서버 프로세스 환경
- 메모리 내 공유 상태 저장소 사용
- TCP 기반 연결
- 텍스트 기반 요청/응답 프로토콜 사용

다음 항목은 현재 baseline 범위에 포함하지 않는다.

- 다중 서버 복제 및 합의
- 사용자 인증 및 권한 관리
- 외부 금융 시스템 연동
- 네트워크 레벨의 완전한 end-to-end exactly-once 전달 보장
- 고급 비동기 I/O 모델 자체를 목표로 한 설계

baseline의 목적은 최적화보다 먼저 correctness, 안정성, 복구 가능성을 검증하는 것이다.

---

## 8. 검증 기준

이 요구사항은 다음 기준으로 검증 가능해야 한다.

- `show`, `buy`, `sell` 기본 기능이 동작하는가
- 동시 요청 상황에서도 stock 수량이 음수가 되지 않는가
- 실패 요청이 상태를 변경하지 않는가
- 성공 응답과 실제 상태가 일치하는가
- 서버 재시작 후 커밋된 상태가 복구되는가

---

## 9. 한 줄 요약

이 시스템은 동시 요청 환경에서도 공유 stock 상태를 안전하게 관리해야 하며, 성공 응답한 상태 변경은 유실되지 않고 읽기 및 재시작 이후에도 일관되게 유지되어야 한다.
